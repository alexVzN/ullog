# Contributing to ullog

Thanks for taking the time to contribute. ullog is a small, focused embedded
C++ library — patches that keep it that way are especially welcome.

## Building and testing

ullog targets C++17 and is exception-free / heap-free. Tests use GoogleTest
and run on a host compiler; the library itself is cross-compiled for ARM in CI.

```bash
cmake -S . -B build -G Ninja -DULLOG_BUILD_TESTS=ON
cmake --build build --parallel
ctest --test-dir build --output-on-failure
```

To cross-compile for ARM Cortex-M locally:

```bash
cmake -S . -B build-arm -G Ninja \
      -DCMAKE_TOOLCHAIN_FILE=cmake/arm-none-eabi.cmake \
      -DARM_CPU=cortex-m4
cmake --build build-arm --parallel
```

## Code style

Formatting is enforced by `clang-format` (version 19+). Run before pushing:

```bash
./bin/run-clang-format.sh    # format in place
./bin/check-clang-format.sh  # CI-equivalent check, non-zero on drift
```

## Submitting a pull request

1. Open the PR against `main`.
2. Keep changes focused — one logical change per PR.
3. Make sure CI is green: host builds (C++17/20/23), all four ARM Cortex-M
   targets, and the format check.
4. The PR title becomes the squash-merge commit message — write it accordingly.
5. Add one label so the PR lands in the right release-notes section (see below).

## Labels and release notes

Release notes are generated by GitHub's native "Generate release notes" button,
configured by [`.github/release.yml`](.github/release.yml). PRs are grouped into
sections by their label, and the PR title becomes the line (we squash-merge, so
title = changelog entry). Pick one:

| Section in the notes | Label |
|---|---|
| Breaking changes | `breaking` (or `semver:major`) |
| Features | `feature` (or `enhancement`) |
| Bug fixes | `fix` (or `bug`) |
| Performance | `performance` |
| Documentation | `documentation` |
| CI & tooling | `ci`, `chore` |
| Other changes | *(no matching label)* |

PRs labeled `ignore-for-release` or `dependencies`, and anything authored by
Dependabot, are excluded from the notes.

## Reporting issues

- Bugs and feature requests: GitHub Issues.
- Security vulnerabilities: see [SECURITY.md](SECURITY.md) — do not file public issues.
